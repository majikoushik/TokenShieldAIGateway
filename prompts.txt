Product Summary:
TokenShield AI Gateway is an AI FinOps and model-routing gateway for enterprises. It helps companies control LLM cost by routing AI requests to the right model tier: cheap, standard, or premium. It also provides budget enforcement, usage tracking, model governance, API key management, audit logs, and an admin dashboard.

Technology Direction:
- Frontend: Next.js + TypeScript + Tailwind CSS + shadcn/ui
- Backend: .NET 8 Web API
- Database: PostgreSQL with Entity Framework Core
- Hosting target: Azure Container Apps
- API Gateway: Azure API Management
- Secrets: Azure Key Vault
- Observability: Application Insights + Azure Monitor + Log Analytics

The product-spec.md file should include the following sections:

- AI token cost is unpredictable.

- Enterprises lack centralized visibility into AI spend.
- Companies need governance over which model can be used by which application/team.
- AI providers may fail or become expensive.
- There is a need for fallback, audit logs, and budget control.

## 5. Same-Tier Fallback

If initial provider call and retry fail:
1. Find other eligible models in the same selected tier.
2. Exclude the failed model.
3. Only use:
   - active tier
   - enabled TierModel
   - active AiModel
   - active Provider
4. Order by TierModel.Priority ascending.
5. Try each eligible model until one succeeds.
6. Rebuild ModelRequest for the fallback model.
7. If one succeeds:
   - return success
   - FallbackUsed = true
   - FallbackFromModelName = initial model
   - FallbackToModelName = successful model
   - FallbackFromTierName = same tier
   - FallbackToTierName = same tier
8. If all fail:
   - proceed to cross-tier fallback if enabled.

---

## 6. Cross-Tier Fallback

If same-tier fallback fails and cross-tier fallback is enabled:
1. Use configured fallback tier order.
2. For selected tier:
   - premium → standard, cheap
   - standard → cheap
   - cheap → no fallback
3. For each fallback tier:
   - find active tier by name
   - find eligible models ordered by TierModel.Priority
   - try models until one succeeds
4. If one succeeds:
   - return success
   - FallbackUsed = true
   - record from/to tier and model
5. If no fallback succeeds:
   - return failure result with safe error message.

Important:
Cross-tier fallback may affect quality. Include this in response metadata and logs.
